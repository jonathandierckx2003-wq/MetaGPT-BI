Chapter 4: Results: Designing an "Agentic BI" Architecture
4.1 Extraction of Source Patterns (MetaGPT)
This first phase, is about 'dissecting' the source domain, which is the MetaGPT architecture. In their research, Hong et al. (2024) demonstrate that complex coding, software engineering and system-building tasks are solved much more effectively when handled by a team of specialized AI agents. An important precision to make is that MetaGPT is not just a hardcoded application, but rather a multi-agent collaborative framework. To prove its effectiveness the authors instantiated this framework with their "First AI Software Company" agentic system case. The framework was thus initially conceptualized and tested for use within a software engineering context, but they mention that "you can use this multi-agent framework in your own scenario to build your own application", and even provide multiple tutorials, additional use case agents (e.g. Data Interpreter Agent), etc. to use and build own new application scenarios (Deepwisdom, n.d.). Their developed "Software Company" agentic system basically mimics the functioning of a real-life software development company, by assigning distinct human-like roles to the agents.
Through the extraction process, 4 main mechanisms from the MetaGPT framework were identified as essential to enable governed multi-agent collaboration. These four core components will serve as a 'blueprint' for the transposition of the architecture into the BI domain. The are the following:
1. Role Specialization: The assignment of distinct, human-like personas (with each specific skills, goals, and constraints) to individual LLM-powered agents in the system.
2. Standard Operating Procedures (SOPs): Predefined, strict, sequential workflows and guidelines that tell agents exactly which tasks they must do, and how and when to perform these tasks.
3. Inter-agent Communication & Memory Mechanisms: A 'publish-subscribe' shared message pool for communication between agents, to ensure that each of them works based on the correct, structured information it needs with no unnecessary noise information.
4. Iterative Feedback and Self-Correction Mechanism: Built-in review mechanisms that allow agents to check, debug and refine their outputs based on auto-
20

mated testing, and to record debug experiences in their memory to improve future performance.
The following sections provide more details about each of these 4 components, by exploring how they function within the original Software Company system of MetaGPT.
4.1.1 Role Specialization
In the same way that a real-world software development process involves various complex tasks for which the collaboration of different people with different expertise is needed, this is also how the agent collaboration is structured in MetaGPT. A specific role with specific skills and expertise is specified and assigned to each agent. Each of them has a specific profile or persona, name, goal, set of constraints, a set of executable actions and/or tools and a specific context. An example of these attributes for the Product Manager Agent is visible in Figure 7. All of these agents behave based on the ReAct (Reasoning & Acting) paradigm introduced by Yao et al. (2022), which means that they follow an iterative loop of "Thought  Action  Observation". They reason plan an action, then execute it, then observe its result, and use this observation to inform the next 'Thought'.
From an implementation-directed point of view, this role mechanism is realized in MetaGPT based on two main base classes the 'standard' Role class and the more extended RoleZero class. This second one, is a specialization of the Role class and thus inherits and extends its working. Both support the ReAct paradigm (with a think() and a act() method), but they differ in how this loop is implemented. In the Role class, the ReAct loop is executed with a max react loop of 1 by default, which means that the agent performs only one cycle and then stops. To determine the agent's next action, three execution options are possible: 'REACT' (the LLM is shown a list of pre-registered actions, thinks and chooses one of these), 'BY ORDER' (actions are executed sequentially in a fixed, defined order), or 'PLAN AND ACT' (the LLM first constructs a plan of multiple sequential actions based on the shown list, then executes it).
In contrast to this, in the RoleZero class only the 'REACT' mode can be used and the max react loop is set to 50 by default. This enables longer autonomous reasoning loops. Instead of selecting from a fixed action list, the think() method assembles a prompt (containing role instructions, available tool schemas, plan status and recent memory) and calls the LLM to decide based on this which tool or command to use next. Additionally, unlike Role, RoleZero re-observes the Environment at every iteration, to see if there is new information to adapt to mid-task. It also includes a humanin-the-loop fallback possibility for when the limit of actions is reached. Another extra method it posesses is quick think(), which does a pre-classification of the request into "QUICK", "SEARCH" or "TASK" before entering the ReAct loop, in order to use a more lightweight response path when the full ReAct cycle is not needed. Finally, RoleZero also supports optional long-term memory, which is not present in the Role class.
Lastly, it is also worth mentioning that the RoleZero class posesses a use fixed sop attribute, that, when set to True, makes the Object fall back to the standard inherited Role behavior. This thus makes both 'modes' available within one same role definition,
21

and switchable according to programmer needs.
To illustrate this, in the Software Company system, the Role Specialization mechanism is used to define the following roles :
· Product Manager: Analyzes requirements and writes the Product Requirement Document (PRD).
· Architect: Translates the PRD into technical system designs and API structures.
· Project Manager: Breaks down the architecture into sequential coding tasks.
· Engineer: Writes the actual Python/software code based on the tasks.
· QA Engineer: Reviews the code, writes tests, and sends it back if it fails.
4.1.2 Standard Operating Procedures
To prevent hallucinations, and unstructured collaboration between the different agents, the MetaGPT framework uses Standard Operating Procedures (SOPs) so that all agents work in a sequential, rigid way, enabling them to follow a strict and streamlined workflow. The authors define SOPs as "the established procedures that govern agent behaviors and interactions, ensuring orderly and efficient operations within the system." (Deepwisdom, n.d.). In essence, these SOPs are the strict definitions of which tasks each agent must perform, in which order, and with which inputs and outputs, received from and delivered to which agent. To make the agents adhere to it, they are encoded into the prompts that define the instructions for each role. Figure 6 illustrates how real-world human team software development SOPs are also used to structure the MetaGPT agent collaboration. As can be seen on the figure, in the Software Company system, the general SOP that is followed entails 7 steps from Planning to Acceptance check, that are performed sequentially by the 5 agents that where defined previously.
For instance, in the Software Company system, this is the main description of the SOP governing the Product Manager agent's process: "The Product Manager generates a Product Requirement Document (PRD), as detailed in the specified documentation. This document encompasses goals, user stories, competitive analysis, requirement analysis and requirement pool. Additionally, a competitive quadrant chart is produced. Subsequently, these documents and charts are handed over to the architect for system design." And this operating procedure is encoded with more details and precision inside the code defining the Product Manager role, as illustrated in Figure 7. On line 36, we can see that the agent acts based on an 'instruction' attribute, which is given by "PRODUCT MANAGER INSTRUCTION". As it can be observed in Figure 8, this variable contains a concatenation of "ROLE INSTRUCTION" and "EXTRA INSTRUCTION". These two variables contain the detailed instructions prompt for the Product Manager agent; the first one is a general insruction inherited from the RoleZero base class that defines the inner ReAct working for all agents, and the seond one is specific to each designed agent. However, it important to note that this instruction attribute is in fact a role-level prompt meant to 'shape' the agent's persona. It is a frame for the agent's general behavior and operating mode. The other specific execution instructions are implemented separately in prompts at the action level, inside

And then ultimately, when an agent is working, these two layers of prompts are sent with the LLM call, but they serve different purposes: the role-level instruction defines how the agent generally behaves, while the prompt on the action level defines what it concretely does when executing a given task.
While Actions define the procedural steps of the SOP, the MetaGPT framework uses Tools to extend an agent's functional reach. While both could initially be confused, Tools are different from Actions in the sense that they are external, modular functions that an agent can call during an execution phase. While an Action represents a more reasoning-related task (e.g., 'Write Code' or 'Analyze Requirements'), a Tool is what provides the agent the technical capability to interact with the outside world. For instance, as can be seen on Figure 7, the Product Manager possesses a tool called 'Browser' which allows him to browse websites and find information on the Internet. These Tools allow the agents to work by not only relying on the LLM's internal knowledge, by grounding their work in real-time data, but above all they allow the agents to interact with and use systems, services and tools that are external to the agentic system.

4.1.3 Inter-agent communication & memory mechanisms
Unlike other LLM-based multi-agent frameworks that use unconstrained, unstructured natural language to let agents communicate, MetaGPT relies on a structured communication approach to avoid the distortion of original information after multiple rounds of communication, and to ensure that each agent works based on the correct information and knowledge it needs. Each agent generates specific outputs based on its SOP instructions and delivers them to the next agent in the workflow through structured outputs like documents and diagrams.
Instead of using one-to-one recurring communication of outputs between agents which is common in other frameworks but inefficient, the MetaGPT framework uses a Publish-Subscribe mechanism. Generated information is stored into a global "message pool" shared with all agents. Each agent can publish their structured messages and outputs in this pool, and can also directly access information published by other agents that it needs for its own tasks. To make sure distraction by irrelevant information is avoided, an agent only "subscribes" to the specific information it needs for its tasks, based on role-specific interests defined in their role profiles and instructions. Figure 9 illustrates this communication mechanism. It shows how the Engineer agent uses coding task specifications from the Project Manager, as well as the PRD generated by the Product Manager, which are both published in the message pool, to perform his coding tasks.
In MetaGPT, each agent (or Role) also maintains its own local memory. This memory is represented by a list of "Message" objects observed by the agent, that are stored in a "Memory" class object for future retrieval. These memories can be called upon when needed, for example to provide context in an LLM call, or to guide next actions. This retrieval mechanism is based on recency, with a 'k' parameter in the get memories() function that sets the number of most recent memories to return, and that returns all memories by default if k is set to 0. This memory mechanism corresponds to "Short-term memory", but more advanced memory retrieval techniques could also be used and are supported by the framework. More specifically, as mentioned previously, the RoleZero class includes an optional RoleZeroLongTermMemory module that enables the use of persistent, vector-based memory retrieval across tasks. How-

4.1.4 Iterative feedback and self-correction mechanism
In programming tasks, processes of optimization and debugging are common. To enable this, the MetaGPT framework incorporates an executable feedback mechanism that will run iterations to optimise the generated code. This is illustrated in Figure 9. As it is depicted, the Engineer agent first generates code based on received PRD and design information. Then it tries to execute this code with corresponding test cases. At each execution, executable feedback is stored into the agent's memory. Based on obtained execution results, it can then continue with the next coding tasks if the test is passed. If not, the Engineer debugs the code using its own execution history and debugging memory in addition to the original PRD and design information. These iterations go on until the test is successfully passed or when a maximum of 3 retries is reached.
In addition to this executable feedback mechanism, the framework also includes a QA Engineer role. This agent is specifically responsible for reviewing and testing the code generated by the Engineer. He takes care of the final quality and acceptance check of the generated application, and sends it back if expected quality is not met.
In summary, this extraction phase shows that MetaGPT 's software engineering agentic architecture relies mainly on the following four pillars: 1) strict role personas with specific goals, actions and/or tools, 2) sequential, strict document-driven SOPs,
26

3) a publish-subscribe communication mechanism, and 4) executable iterative selfcorrection loops.

The term "transposition" is used across the following chapters to refer to the practical application of this "exaptation".
4.2 Conceptual Transposition (Transformation to target domain)
To transpose the agentic Software Engineering architecture towards an agentic BI architecture, we first need to define the precise BI workflow we aim to automate. This will allow the definition of agents to perform the different steps, as well as a general SOP to structure their collaboration, and individual operating procedures, profiles etc. for each agent.
As established in the literature review (cf. section 2.1.2), the BI workflow is a complex multi-phase process. Unlike most frameworks and systems developed in existing literature, which focus on automating specific tasks or isolated parts, this transposition attempts to design an architecture that covers the entire "back-end" BI solution development workflow. The workflow that is targeted for this agentic architecture's SOP thus goes from the initial Requirements Elicitation phase, to the Dimensional Modeling phase and finally to the data transformation pipeline phase. In previous work, such as DataLab (Weng et al., 2025) that was presented in the Related work section (cf. section 2.2), the ultimate visualization, dashboarding and reporting phases are also considered in the agentic workflow. In these projects, this is operationalized by using Python libraries, "BI-as-code" or other similar ways, to provide the wanted visualizations using code. While this integrates well in such systems where the main objective is to produce BI analysis 'on-the-go', it doesn't really allow to produce durable visualization/reporting solutions that can be used like a daily information retrieval application. Additionnally, in real enterprise environments, these phases of BI development are mostly done using mainstream proprietary BI platforms like Microsoft PowerBI, Qlik, Tableau, etc., which until recently where more complex to bring in interaction with automated agents. As of today, new protocols like MCP have been brought on the market by big BI actors like those, which allow to interact with their BI platforms directly from within LLMs and AI agents. However, the choice was made to not include those last phases in the scope of this transposed architecture, to keep the focus purely on the "back-end" development of the BI data architecture.
Another scoping decision that is important to mention concerns the temporality of execution of the designed agentic workflow. In the original MetaGPT system, multisession, incremental project workflows are supported, which basically allows human users to come back to existing projects or existings parts of objects and work further from there or request updates to previously created artifacts (e.g. adding a new feature to an existing PRD). This is implemented through dedicated branching logic inside the code scripts that define an agent's action and use a versioned filesystem to keep
27

artifacts across sessions. However, this adds significant complexity and is therefore deliberately left out of scope in this transposed BI architectured. The designed agentic workflow is thought as a one-shot pipeline in which each session starts from a new user request and produces a complete, BI back-end development from requirements elicitation to a working DWH. While this simplification reduces architectural and implementation complexity for the PoC, in the real-life enterprise domain, BI systems actually evolve continuously over time (e.g. addition of new KPIs, extension of existing tables, data source change, etc.). The transposition of MetaGPT 's incremental project mechanisms into the BI domain would therefore be very interesting, but is left as a relevant direction for future iterations of the DSR Design cycle, as further discussed in section 5.2.
Additionally, while the author bases this transposition on widely accepted definitions of a typical BI workflow and architecture as presented in the Literature Review (cf. section 2), important is to acknowledge that this remains a not fully neutral choice. As many previous works showcase, including work of previous KU Leuven master students (Van Weyenbergh, 2024), in the BI world a wide range of different architectures and workflows exist, and the one that is applied by the author in this thesis is just one of them. Another author could have chosen to follow a BI workflow differing in some aspects (e.g. different sequence logic, Data lake instead of DWH, etc.), which would ultimately lead to differences in the designed architecture, or would require a more flexible agentic system, capable of adapting to different BI workflows and architectures on user demand.
The following subsections execute the conceptual transposition from extracted concepts to the target domain by applying the four transformation sub-steps defined in section 3.3.
4.2.1 Role and Output Artifact Mapping
To achieve the automation of these steps, the 5 default software engineering roles used in the original MetaGPT framework (Product Manager, Architect, Project Manager, Engineer & QA Engineer) are systematically mapped onto their 'functional equivalents' within the BI development workflow. This results in the following 5 BI-specific roles:
· BI Requirements Analyst (transposed from MetaGPT 'Product Manager'): Conducts conversational requirements elicitation with the business user as well as source-driven elicitation (based on the source data it has access to) to define KPIs, requirements, etc., outputting a formal Business Requirement Document (BRD).
· BI Data Modeler (transposed from MetaGPT 'Architect'): Based on the BRD, and the source data, decides on the best data schema (star, snowflake, or starflake) to use and translates it into a conceptual and logical dimensional model.
· BI Solution Architect (transposed from MetaGPT 'Project Manager'): Bridges the logical model to physical execution by defining the external tools and services to use, the technical project structure and a sequential task list to execute to create the needed DWH.
28

· BI Analytics Engineer (transposed from MetaGPT 'Engineer'): Performs the actual actions against external tools (data sources, data transformation tools, DWH hosting systems) to perform ETL (Extract-Transform-Load), build the data pipelines and deliver a ready-to-use DWH.
· BI QA Engineer (transposed from MetaGPT 'QA Engineer'): Reviews the generated DWH, executes automated data and quality tests, and routes execution errors back to the BI Analytics Engineer for self-correction.
This performed methodological mapping of roles is based on the principle of functional similarity. This means that each identified software engineering role is transposed to a BI role that performs a similar operational function within the BI workflow. Because these two different domains share multiple functional similarities and analogies in their workflow, a mapping between both can be established. To give this mapping a more concrete academic base, it can be grounded into the concept of "analogical structure-mapping" as defined by Gentner (1983). According to this established method, mapping objects across different domains must be based on relational similarities rather than just matching the individual objects or basing it on superficial features. In this context, not only do the individual roles share functional similarities, but the overarching workflow also preserves relational similarities, which aligns the mapping with Gentner's analogical structure-mapping theory. For example, the Product Manager's function is eliciting requirements and defining the product's goals, which is functionally similar to the BI Requirements Analyst's role of eliciting business requirements and defining KPIs. But their relational structure is also similar in their respective workflow: just as the Product Manager passes requirements to the Architect for system architecture design, the BI Requirements Analyst passes the Business requirements to the BI Data Modeler to define the BI data model and architecture.

4.2.2 Standard Operating Procedure (SOP) Redesign
Now that the new BI roles and their corresponding output artifacts are established, the overarching workflow governing their collaboration must also be redefined. In the original MetaGPT framework, Standard Operating Procedures (SOPs) are not just procedures dictated by a central, main control script. Instead, the framework relies on an event-driven, 'decentralized' way of coordinating the collaboration. Agents continuously observe a shared virtual "Environment" (the global message pool that was described in section 4.1) and trigger their own tasks only when they detect the publication of a prerequisite artifact by an upstream role.
To effectively adapt and transpose this mechanism into the target domain of BI, the sequential workflow through which the agents go, is rewritten to reflect the followed BI development workflow. By doing so the global SOP is redesigned, and forms a strict chain of dependent tasks that define exactly which inputs are required to trigger each agent, and which resulting outputs they must necessarily publish back in the shared pool to allow going further along the workflow.

4.2.3 Persona and Prompts Engineering
To make the transition from a high-level, theoretical SOP to an architecture that is really executable, the profile and behaviour of each agent must be strictly formalized. This is because LLMs are inherently probabilistic models meant as general-purpose assistants, and prone to hallucinations or inconsistencies. Therefore, they require rigorously engineered instruction prompts, with operational context and constraints, if we want them to function and act precisely as the enterprise professionals we want (Hong et al., 2024).
In line with step 3 of the Transformation phase defined in section 3.3, this section defines the profile, the specific attributes (name, goal, constraints, tools/actions) and instructions for each agent of the agentic BI system. These prompts definitions will sub-
34

sequently be used to initialize the agent classes in the transposed codebase. To avoid overflowing this section of the thesis and keep text explanations clear for the reader, the full instruction and action-level prompts are given as an illustration for Agent 1 only. For the following agents, only text design explanations are provided in this section of the thesis, referring to the full prompts for each agent, which are retrievable in Appendix B.
As mentioned in section 3.3, to construct effective prompts, the author relies on those used in the MetaGPT original Software Company code, but also on the Prompt Engineering Guide by DAIR.AI (2026). It defines 5 general principles that are followed closely, as well as a number of established prompting techniques, of which some are applied across the agents of this architecture.
First of all, the agent instruction prompts are mostly designed as zero-shot prompts, relying mostly on detailed role instructions and explicit output format specifications instead of real worked out examples. For some specific parts of the designed prompts a little bit of few-shot prompting is used (e.g. to provide a set of example query/analyses that could be part of a BRD). In future iterations of this architecture design, it would be beneficial to try using more few-shot prompting by including more clear examples in the instruction prompts and see if this improves the generated outputs.
In several of the constructed prompts, the Meta-prompting technique is also used. When using this technique the focus is put on the structural and syntactical aspects of the response we expect from the LLM, by providing a clear wanted format or structured template. This is especially used in the prompts of this agentic architecture that define the artifacts agents need to produce and their expected format.
Within each agent, a Chain-of-Thought decomposition technique is also applied: rather than telling them to produce an output artifact directly from just a vague input, each instruction prompt breaks the task into multiple reasoning steps that the agent must go through before producing the final output.
At a higher system-related level, the Prompt Chaining technique is also put in place, by the multi-agent SOP workflow itself. Indeed, in this architecture the structured output artifact of each agent becomes the input context for the next one, thus breaking the "BI back-end development" task into subtasks with each an own agent and prompt. This helps achieve better performance controllability and reliability.
And finally, the iterative self-correction loop that is present between the BI QA Engineer and the BI Analytics Engineer is a direct application of the Reflexion technique. The Analytics Engineer agent observes the result of its action in the Feedback Report received from the QA Engineer, reflects on the cause of failure and then uses that to improve its next attempt.
The following subsections apply these principles consistently across all five agents.
4.2.3.1 Agent 1: BI Requirements Analyst
In the original MetaGPT framework, the Product Manager role is responsible for analyzing the user's initial (often vague) prompt and generating a Product Requirement Document (PRD).
In the BI development domain, accepting a one-shot prompt to build a BI architecture would most likely lead to failure, as business users rarely formulate their exact
35

analytical needs perfectly on the first try and in one sentence. Instead, we really need an agent that can iteratively elicit the complete requirements and needs of the user, by engaging conversation and by analyzing the source data that it has access to. However, this requires a well-thought adaptation, as in the original MetaGPT Software Company system the ProductManager receives a single, one-shot UserRequirement message and immediately writes the PRD from it. In fact, this is because the default used Environment loop in MetaGPT is designed for sequential, non-interactive pipeline execution, where agents act as a reaction to messages from other agents placed in the Environment shared message pool, and not from humans (except for the initial request message).
Therefore, to enable conversational elicitation within this architecture, the BI Requirements Analyst agent is built using the previously defined RoleZero class, in order to use its built-in reply to human command. This will allow the agent to put his ReAct loop on 'pause' at any point, to send a question or specific clarification request back to the human user, and then resume execution once an answer is received. And all of this, while simultaneously retaining the conversation history in his memory accross conversation turns. By doing so, the agent will iteratively ask structured elicitation questions (covering all elements of the BRD) until he determines that sufficient information has been gathered, and at that precise point he writes the final BRD and publishes it in the shared message pool, which will trigger the next agent in the workflow.
It is worth noting that MetaGPT also provides a second, distinct human engagement mechanism : the is human=True parameter. When turned on, it replaces an entire role with a human operator that is then entitled to his responsibilities in the workflow. While this could be used in a future iteration to insert a human validation gate between BRD publication and the triggering of the BI Data Modeler, allowing a business stakeholder to formally approve requirements before technical development begins, it would require to add an extra "BRD Validator" intermediate human role and is thus left outside of the scope of this thesis for simplicity. In specific cases, this parameter could also be used to replace some of the encoded agents by human agents (e.g. in a case where an enterprise would automate everything except the Requirements elicitation part).
Next to conversational requirements elicitation, the BI Requirements Analyst is also designed to autonomously inspect available data sources. To do so, a dedicated "DataSourceInspector" Tool is implemented to support connections to multiple types of databases and flat data files (e.g. a CSV file). Using this, the agent can extract the structure (table names, column names, data types and row counts) of each available source. Connection details for the source (e.g. API credentials) are asked to the human user through conversation and the retrieved schema information is then stored in the agent's memory, which will allow him to ask more informed elicitation follow-up questions based on this before producing the final BRD. These two elicitation capabilities are aligned with both the analysis-driven and source driven requirements specification approaches described by Vaisman and Zima´ nyi (2014) as discussed in the Literature Review (cf. section 2). This brings a big functionality extension compared to the original one-shot requirements MetaGPT pattern.

4.2.3.2 Agent 2: BI Data modeler
In the original MetaGPT Software Company System, the Architect agent makes the bridge between requirements and actual software engineering, by translating the received PRD into a software system design and architecture (data structures, APIs, file structures, program calls, etc.).
In this transposed BI architecture, the BI Data Modeler takes on the functionally 'equivalent' role. He takes the BRD, that was published in the shared message pool by the BI Requirements Analyst, as input and transforms this into a dimensional architecture design for the future Data Warehouse that will serve the BI solution. More concretely, this includes choosing the most appropriate dimensional schema type (star, snowflake or starflake), identifying what are the facts, dimensions, measures and hierarchies that are needed to answer the requirements of the BRD and also generating both a conceptual and a logical data schema for the envisioned DWH.
Some design decisions that were iteratively made to transpose this role from the original Architect agent are worth noting, before continuing to the agent's attributes and prompts definitions.
First of all, concerning the way this agent is guided for his dimensional schema type decision, multiple different approaches have been considered. A first option was to base the agent's decision-making in a RAG module, in which he could retrieve relevant explanations and examples from dimensional modeling literature (e.g. (Kimball & Ross, 2013), (Vaisman & Zima´ nyi, 2014)) inputted into it. The MetaGPT framework does actually support such an integration natively through its RAG module where a SimpleEngine class retrieves info in a vector store created from documents, as illus-
42

trated in the MetaGPT documentation examples (FoundationAgents, 2024). However, two practical considerations led to not using this approach in the current architecture. Firstly, as these dimensional modeling sources are copyrighted, published books, ingesting them into a vector store for use in enterprise-related contexts forms a form of redistribution that go along with significant risks if the RAG system reproduces coprighted content freely. Secondly, the schema-type decision is in essence a bounded problem with only a small number of criteria to take into account. Using RAG retrieval only to get to these bits of information would arguably add infrastructural complexity into this agentic architecture design without delivering really proportional benefits. Therefore, the approach that was finally chosen in this thesis research is to encode specific decision criteria directly into the agent's instruction prompt, derived from the established dimensional modeling literature by Kimball and Ross (2013) and Vaisman and Zima´ nyi (2014). The use of RAG over a specialized knowledge base is however still left as a possible improvement in future iterations of the Design cycle of the followed DSR process, as mentioned in section 5.2.
A second key design choice concerns the modeling notation used by the agent to produce the conceptual and logical schemas. Vaisman and Zima´ nyi (2014) have explored this subject extensively and even propose their own MultiDim conceptual model notation which is specifically tailored to DWH multidimensional design. However, they also acknowledge in their book that the 'classical' Entity-Relationship (ER) model remains the most widely used conceptual model for database design and is broadly seen as a standard across both academia and industry. And for the logical level, they adopt the Relational model which is also seen as the dominant standard for logical design of databases. As Vaisman and Zima´ nyi's own MultiDim notation requires specific graphical elements and symbols that are often not inherently provided by today's code-based diagram drawing languages and given that current LLMs are better at generating standard ER and Relational models due to their wide presence in their training data, the ER model is used as the conceptual model and the Relational model for the logical schema in this transposed architecture. To produce both these schemas, the system relies on Mermaid, and more precisely 'erDiagram' code blocks (Sveidqvist et al., n.d.). This mirrors the pattern of the original MetaGPT Architect role, who first produces a markdown system design followed by Mermaid 'classDiagram' and 'sequenceDiagram' visualizations. Other code-based diagram drawing languages or tools such as PlantUML, or even DBML/DBDiagram (which are specifically addressing database modelling) could have also been used for this (Holistics Software, 2026; Roques, 2026). However, since Mermaid was already used and implemented in MetaGPT 's source code and possesss the erDiagram' code blocks that should be able to produce the needed schemas, this implementation is used for simplicity. If the produced schemas appear to be unfit or not sufficient, Mermaid could be replaced by one of the aforementioned tools in future design iterations.
A third important decision to notice is about also giving the BI Data Modeler agent direct access to the available data sources, in addition to the access that is already provided to the BI Requirements Analyst. This would allow the BI Data Modeler to verify directly that dimension and fact tables he proposes can actually be constructed from the source data and could help him decide on data types, key candidates, etc.
43

However, the choice was finally made here to not give this access. This is because it would break the document-driven SOP principle that was presented in section 4.1.2 as one of the core elements of the MetaGPT architecture. Therefore, instead of a direct data source access, the BI Data Modeler relies exclusively on the BRD that was published by the BI Requirements Analyst, which must include the data source schema summaries in Section 6 of that BRD document. This preserves the structured publishsubscribe communication workflow defined in the original framework, prevents different interpretation of source data structure (only one single source of truth) and keeps each agent focused on its own core responsibility and subtasks.
And finally, about the task execution structure, the BI Data Modeler agent is designed to work with one single Action class, which groups the production of all three artifacts (the dimensional model specification, the conceptual ER schema and the logical Relational schema) in one execution. This, again, perfectly mirrors the inner working of the original MetaGPT WriteDesign Action class that also generates a system design in markdown format, along a class and a sequence diagram, all in one single coordinated execution. Thereby, using this consolidated structure instead of splitting the execution across three separate Action classes stays consistent with the DSR exaptation methodology followed in this thesis, which aims to keep the adequate system patterns of the source framework while adapting them to the target domain. 4.2.3.3 Agent 3: BI Solution Architect
In the original MetaGPT Software Company System, the Project Manager agent takes the system design produced by the Architect as input and produces a coding execution plan based on this. This takes the form of a structured list of sequential coding tasks in a JSON file that the Engineer will subsequently execute one by one.
In this transposed agentic BI architecture, the BI Solution Architect plays the functionally similar role. He takes the Dimensional Model Specification, the Conceptual Schema and the Logical Schema that BI Data Modeler produced as inputs and then translates these into a structured 'DWH Technical Execution Plan' that defines exactly all the tasks that must be performed to create the envisioned DWH that the human user needs. He also specifies the order in which these tasks must be performed, which is very important to avoid fails (e.g. a data transformation task would not be possible to execute if the connection with the data source is not first established).
As for the previous agents, the performed transposition to this agent's working required several design decisions, of which some should be made clear before continuing to the definition of its attributes and prompts.
A first important observation concerns the BI architecture that is utilized in the workflow supported by the designed system. As presented in the literature review (cf. sections 2.1.3 & 2.1.2), the BI landscape has progressively moved from the traditional Extract-Transform-Load (ETL) pattern (where data is transformed outside the DWH before being loaded into it), towards the more modern approach of Extract-LoadTransform (ELT), where raw data is loaded into the DWH first and then transformations are applied to it inside the DWH. In this transposed agentic architecture, the choice was made to assume produced solutions exclusively use the ELT architecture for two different reasons.
Firstly, ELT is today's dominant approach in modern enterprise BI infrastructures and is used in many transformation tools like dbt or Airbyte. It is widely considered as a standard for cloud-native data stacks.
Secondly, ELT operates with a clear separation between ingestion and transformation phases: all raw data is first loaded into temporary tables within the DWH by an ingestion tool, and then a transformation tool applies the needed transformations to populate all the final dimensional tables.
This clean separation maps very well onto a multi-task agentic workflow like this one because each task can be then assigned easily to the appropriate tool with explicit dependencies between them. On the other hand, ETL has more blurry boundaries between ingestion and transformation phases with separated transformation followed by ingestion for each table. This requires using transformation tools followed by ingestion tools iteratively for each table and makes task decomposition more complicated. The case where a user would explicitly want to use a traditional ETL pipeline is therefore deliberately left out of scope for this thesis and acknowledged as a possibility for future imporovement in section 5.2.
A second important design decision concerns the role-level instruction prompt for this agent. In the original MetaGPT framework, the Project Manager role is very simple at the role-instruction level. In the Software Company system, the instruction attribute
47

for this role contains just a single one-line instruction ("Use WriteTasks tool to write a project task list"), and the actual task-breakdown instructions are almost entirely given inside the action-level prompt of its WritePlan Action class. This is sufficient in the original software engineering context because LLMs often handle software task breakdown quite well based on their training data alone, and because the available task type categories in the original framework are small and well-known.
In the BI domain however, there is significantly more complexity in in the planning of the different DWH creation tasks. It requires to discern multiple diverse task categories (e.g. DWH instantiation, source connection setup, raw data ingestion, data transformation, etc.), with explicit dependencies between them. When automated by an agentic system, it also involves the integration of multiple distinct external tools and humanin-the-loop checkpoints to collect connection credentials for some of these tools. This added complexity brings the need for a more extended role-level prompt than in the original framework, to encode the BI-specific task categories, the task dependency logic and the clear rules for tool selection.
Moving forward, concerning the format of the DWH Technical Execution Plan that this agent produces, it is important to highlight that, as briefly mentioned in section 4.2.1, this artifact is produced as a structured JSON document. This choice is a direct transposition of what is also done in the original MetaGPT system and is a practical necessity to streamline the action that the next agent (BI Analytics Engineer) will undertake. Indeed, given that this one must be able to automatically iterate through the tasks, understand their dependencies before execution, and then dispatch each of them to the appropriate Tool classes (that will call extenal tools) based on their specific task details, a simple markdown document would not allow this easily. Therefore, a JSON list of tasks is produced with each task conforming to the following schema:
· task id: unique identifier for the task within the plan
· dependent task ids: list of task IDs that must be completed prior to this task
· instruction: natural-language description of what the role must do in this task
· task type: one value from a closed list of BI-specific task categories (see below)
· tool: name of the external tool selected to execute the task (when applicable)
· tool args: dictionary of tool-specific arguments needed for execution (when applicable)
The two last fields are not present in the original MetaGPT Project Manager's JSON output, and are an adaptation brought for this transposed architecture. This addition is a direct consequence of the difference between the source and target domains. As in the software engineering context, the Engineer agent operates within a single execution environment (which is typically a Python interpreter and the local filesystem), the task type alone provides enough information to determine how the task should be executed. But, as explained further in text, in the BI domain the Analytics Engineer must also connect to several external tools/systems to execute some of his tasks (e.g. a TRANSFORMATION task could be executed using dbt, or Airbyte or some other tool).
48

The tool field thus specifies the tool that must be used for the task, and the tool args field carries the specific configuration information that is needed to execute the task with that tool (e.g. database connection string, source file path, etc.).

4.2.3.4 Agent 4: BI Analytics Engineer
The BI Analytics Engineer agent is the most operationally complex agent in this transposed architecture. This agent is responsible for the 'translation' of the DWH Technical Execution Plan provided by the BI Solution Architect, into actual executed tasks. To do so, it interacts with multiple external tools and systems for data warehousing, ingestion, transformation, etc. and outputs an Execution Report after operation is complete.
In the original MetaGPT framework, two role implementations exist for the Engineer agent: Engineer & Engineer2. The first one uses the base Role class and a fixed, rigid SOP, while Engineer2 is a more modern rewrite of the role that uses RoleZero as its base class and operates as a more autonomous ReAct agent. Another big difference is that the Engineer role pre-builds a list of WriteCode Actions and then iterates through them mechanically, while the Engineer2 reasons at each step on what to do next, selects tools dynamically from a registered tool set and defines its primary execution method (write new code) directly in the role class itself as a method 'decorated' with "@register tool" rather than in a separate Action subclass. In simpler words, this means that the engineer's 'code writing' ability is built right into the class like a built-in command that an LLM can trigger when needed. The @register tool(include functions=["write new code()"]) line of code, basically makes the specific "write new code()" function defined in the same role definition code script callable, without exposing other internal functions in the script to the LLM.
For this architecture transposition, the choice was made to work from the Engineer2's architectural pattern. Indeed, as the BI Analytics Engineer must be able to dispatch heterogeneous task types to different external technologies/systems, to handle exceptions, to pause execution for credential collection from the human user, and also to respond to corrective feedback from the BI QA Engineer, this wouldn't fit the more static model of the simpler Role base class. In consequence, the BI Analytics Engineer inherits from RoleZero, uses a long instruction prompt that is sent to the LLM at each thinking step of the ReAct loop, and the same write new code() pattern as in the orginal MetaGPT system is applied through the execute BI task() method. This function is defined directly in the BI Analytics Engineer role class and 'decorated' with @register tool, making it callable by the LLM within the ReAct loop. Concretely, execute BI task() acts as a Tool dispatch router: when it is called with a task object from the JSON execution plan, it reads that task's task type, tool and tool args fields and dispatches execution to the appropriate registered Tool class (e.g., DuckDBExecutor for a SCHEMA CREATION task, DbtRunner for a TRANSFORMATION task, etc.). For CREDENTIAL REQUEST tasks, it calls RoleZero.reply to human, waits for the human user's response, and then stores the returned credential or connection instructions in the execution context before returning. After each Tool dispatch, the method returns an execution result (success or failure status, row counts, error messages, test
52

results) which the LLM observes in the next ReAct step in order to decide what to execute next. A key architectural point to understand is that execute BI task() does not itself perform database operations or transformations: those are performed by the individual Tool classes it calls. The LLM is responsible for selecting which task to execute next; execute BI task then routes that task to the correct Tool and returns a result that the LLM can observe before continuing. In section 4.2.4 a deeper explanation is given about the concrete implementation of the Tool classes used by this agent, including how they handle multi-step interactions with external systems that are sometimes needed to execute tasks (e.g. for one single TRANSFORMATION task it might be necessary to connect to a dbt project, then write the needed transformation code, then try to execute that code in dbt and possibily correct it, etc., each needing observation and reasoning between them).
The maximum number of ReAct iterations (max react loop) is also set high as a high number of tool calls will be required to execute a full ELT process.
While this agent is strongly based on how the original framework's Engineer2 functions, 2 specific adaptations needed for the BI domain of this designed architecture should be brought forward. First, unlike the software engineering domain where all final output artifacts are code scripts written into a local filesystem using only one consistent tool, in the BI development domain, multiple different task types must be executed using multiple different external systems (e.g. a connection setup task to connect to source databases, a transformation task to make data transformations via an external system, etc.). As it was defined in section 4.2.3.3, it is the BI Solution Architect that defines the tool that will be used for each task, and that encodes this in the DWH Technical Execution Plan. But, depending on the task type, the BI Analytics Engineer must follow different specific rules, which are given per task type in the role-level instruction prompt.
And secondly, unlike the original framework's agent, the BI Analytics Engineer must be able to pause execution, in order to ask connection details or credentials to the human user (CREDENTIAL REQUEST tasks). To do so, it calls RoleZero.reply to human (which was also used in Agent 1 for conducting an elicitation conversation), it stores the returned credentials or intructions and only then continues execution. This specific interaction pattern has not really an equivalent in MetaGPT 's original Engineer2 agent, which operates entirely autonomously. It thus represents a real domain-specific extension whose need is caused by the reality that in a BI ELT workflow, connection is made with multiple external data systems for which access is often restricted by credentials or connection tokens which must therefore be given to the agent by the human user.
A final design decision worth mentioning here concerns tool extensibility in the presented agentic BI architecture. For the PoC that is made in this thesis research, only a limited set of external tools/systems (DuckDB, dbt, SupaBase, pandas and Airbyte) is implemented in the system. However, a wide range of tools exist in the BI landscape, so to ensure extensibility of the system, it is designed in a way that adding a new tool in future requires minimal changes. Each available Tool is encapsulated in its own Tool class file which is 'decorated' with the previously presented "@register tool" decorator and the file is located in metagpt/tools. To add a new tool to the system, a developer must thus only write its corresponding class file in metagpt/tools, add its name to the
53

role's tools attribute list to make it available to the BI Analytics Engineer. It's also important not to forget to add it's name into the BI Solution Architect prompt (cf. section 4.2.3.3) so that the tool can actually be added as a value in the "tool" attribute of tasks listed in the DWH Technical Execution Plan.

In those cases, the BI Analytics Engineer must read the QA Engineer's feedback, identify failing task IDs and then re-execute only the concerned tasks and their downstream dependents. This allows a targeted correction loop rather than a making the agent perform a full restart of all tasks, which would not be efficient.

55

mechanism by which the LLM is instructed to produce a structured document artifact (BRD, Dimensional Model Specification, DWH Technical Execution Plan etc.). These are fundamentally tasks where some artifact is generated by the agent itself and a prompt template is thus sufficient to guide on doing that.
The BI Analytics Engineer's task however, is quite different as it is a real execution task, where the agent must interact with external systems and tools, not a fundamentally generation task. In this agent's working, the LLM's main role is to decide which tool to call next (based on the execution state and the plan) in order to perform all needed phases of the DWH creation process. Its primary task is thus not to generate a document or artifact (except for the Execution Report produced a the end). Subsequently, the actual tools that are called for each task of the execution plan, are handled by the Python implementation of execute BI task() and the individual Tool classes that are defined. This architectural difference is a direct transposition of how things are done in the original Engineer agent, where similarly there is no central PROMPT TEMPLATE for its writing work but instead, it uses a standalone prompt constructed inside write new code() when tools are called, instead of a high-level prompt template shared across all calls.

One specific artifact production question this brings forward is the following: "How is the Execution Report (which is the only document artifact produced by this agent) written, given that no dedicated Action subclass or PROMPT TEMPLATE is defined for it?". To answer this need, this is is handled directly within the agent's ReAct loop, through Step 3 of the role-level instruction prompt (Cf. Appendix B.4) and implemented into the execute BI task() function itself. In the designed BI architecture, after each call to execute BI task() the result, composed of the task ID, its execution status (complete or failed), a short output summary, and for some tasks also the row counts (DATA INGESTION tasks) or test results (TRANSFORMATION tasks), is returned to the LLM and stored in the agent's memory. When all tasks in the execution plan have been processed, the LLM is instructed by Step 3 of the role-level prompt to synthesize the accumulated results into a structured Execution Report, and to use the Editor tool to save it into the project's docs folder. This way the Execution Report artifact can be produced without needing a separate Action class. This approach is consistent with how the original system's Engineer2 role handles its own output consolidation, and allows to keep execute BI task() as the single execution and reporting authority for this agent. This avoids the need to constuct an extra separate Action subclass to call by the LLM.
4.2.3.5 Agent 5: BI QA Engineer
In the original MetaGPT Software Company System, the QA Engineer agent is the most minimalist role what concerns prompting. Unlike all the other roles in the agentic framework, it has no role-level instruction prompt at all. Instead of this, the agent's
56

behaviour is completely defined by using Python logic. The agent has a very deterministic way of acting: when new code produced by the Engineer agent is observed, the QA Engineer calls the WriteTest action to generate corresponding test files, followed by RunCode to execute them, and then DebugError to fix those that appear to be failing. As explained in section 4.1.4, this is done until tests pass or until the maximum amount of iterations is reached. To decide if it is necessary to loop back and send feedback to the Engineer or not, the text output of RunCode is parsed, sent to the LLM who decides to whom it must be routed (NoOne, Engineer or QaEngineer). The original system's QA Engineer thus in fact produces no document artifact, but uses temporary messages to route back if correction is needed, and terminates when all tests pass.
However, the transposition from this way of working to our Agentic BI architecture is a bit more complex than that. Indeed, in the software engineering context, this routing mechanism works well because the quality criterion we test is just binary and machine-verifiable: either the tested code scripts work or they do not. In contrast to this, assessing quality in the BI development domain entails more than just checking if code is running without errors. A produced DWH can be operational on a technical view while still not being entirely valid on a business perspective (e.g. missing KPIs, fact tables with wrong granularity, dimension that lack some attributes specifically needed to answer a query that was specified in the BRD, etc.). To detect these types of failures, a QA agent that can reason about the content and structure of the produced DWH in relation to the original business requirements is needed. For this reason, the BI QA Engineer agent is designed with RoleZero as base class instead of Role, and with a Validation Feedback Report document as output artifact instead of the one-word routing used by the original framework's QA Engineer role.
Concretely, the designed BI QA Engineer takes the Execution Report as input (which contains instructions to connect to the created DWH), analyzes the created DWH against requirements of the BRD, contents of the Logical Schema and of the DWH Technical Execution Plan, and then produces a Validation Feedback Report that either accepts the created DWH as-is, or flags errors which then trigger the BI Analytics Engineer for correction.
To do so, the BI QA Engineer performs two types of validation checks: structural/technical validation & requirements traceability validation. The first category is about verifying that all the fact and dimension tables that were defined in the Logical Schema actually exist with all correct columns and data types in the DWH that was created, that all tables contain data after the performed ELT process, that all primary keys are unique and that foreign key referential integrity is respected. The second type of validation checks is basically comparing the created DWH with the BRD to verify that every query/analysis listed in Section 4 of the BRD can be satisfied in a BI visualization based on the available fact and dimension tables, that every KPI defined in Section 5 of the BRD can actually be calculated from the available data at the required granularity, and that every data source listed in Section 6 of the BRD has been included in the DWH. While the first type is a bit closer to the testing that is done in the original MetaGPT framework, the second one is a very BI-specific adaptation brought by this transposition that allows the agentic system to ensure that the produced DWH respects the initially collected requirements and that is tailored to the human user's needs.
57

In the original MetaGPT system, the QA Engineer also enforces a hard maximum of correction rounds (by using test round allowed) set to 5. This mechanism is kept in the transposition towards the BI QA Engineer, to ensure that the agent doesn't continue running indefinitely. If this limit is reached without successful validation, the Validation Feedback Report is published under a different name ("FailedValidationReport") but with the same structure highlighting persisting issues, so that the human user can consult it while the BI Analytics Engineer agent doesn't see it, to avoid that he continues to try fixing the issues.

As established in section 4.2.3.4, if any new external tool must be added to those suported by this agentic system this can easily be done by writing a tool class for it, by adding its name in the BI Solution architect's prompt and by registering it in the Analytics Engineer's tools list. 4.2.4 External Tool Integration
As it has been explained in section 4.2.3.4 already, one of the biggest differences between the original MetaGPT Software Company agentic system and this transposed agentic BI architecture concerns the precise execution environment in which the designed agents operate. In the default Software Company case of the original framework, the agents mostly interact within a single homogeneous execution environment: a self-contained local Python interpreter that writes and executes files and code in a local filesystem. While the Tool class of the framework allows to configure external interaction with the outside world, this is only used minimally in the Software Company system, for instance to allow the Product Manager role to conduct web search in order to gather competitive analysis and market data from the Internet.
However, in this transposed agentic BI architecture, letting the agents operate locally only is not an option. Indeed, as established across the previous sections of this chapter, interaction with external tools and systems is a critical need to automate a BI
60

development workflow. The BI Requirements Analyst must have access to the source data systems in order to understand their structure, the BI Analytics Engineer must be able to execute tasks within multiple external systems (a DWH hosting service, a data transformation tool, a data ingestion connector, etc.) to effectively build out a DWH, and the BI QA Engineer must have the ability to query that same DWH to run its validation checks. While the agentic BI systems developed in previous studies like DataLab by Weng et al. (2025), which is working as a unified platform in a notebook environment to provide ad-hoc BI insights, integrate with external databases, they only interact with these external systems to ingest data which is then processed "on-the-go" in the platform's self-contained environment. In contrast with this, the agentic BI system that is designed in this thesis, must not only be able to ingest data from data sources, but also to operationalize its data transformation, and to build a lasting DWH hosted in an external system or database, that can subsequently be used as source for dashboarding and reporting.
Therefore, to allow automation of the DWH creation tasks within this transposed agentic architecture, an explicit external tool integration strategy is needed.
This is made possible by defining new custom Tool classes within the MetaGPT framework, tailored to the agentic BI architecture's needs. As introduced in section 4.2.3.4, to integrate these Tools in the workflow followed by the agents of the transposed architecture, MetaGPT 's native @register tool 'decorator' pattern is used. The code that allows to connect and interact with each needed external system, tool or service, is encapsulated in a dedicated Python Tool class file and saved in metagpt/tools/. Each of these created custom classes is decorated with @register tool, and this has two effects:
1. It registers this new custom Tool class into MetaGPT 's global tool registry under its class name, making it available for integration within role classes, and
2. It makes the documentation strings of all the methods/functions in that Tool class available to the reasoning LLM of agents that listed it in their tools attribute, at runtime.
In simpler words, this means that every added Tool class is documenting itself so that the agents' LLMs know what each tool they have access to does exactly and how it must be called, without needing additional instruction prompts in these agents' role classes. To make one of these Tool classes available for use by a specific agent, its registered name is added as a string to that agent's tools attribute list, like it was done in the agent attribute tables from section 4.2.3.
In the context of this thesis, only a limited number of external tools have been implemented in the PoC (cf. sections 4.2.4.1 & 4.3.1), but in today's BI landscape a wide range of technologies and systems exist (data sources, data transformation tools, data warehousing systems, etc.) that would be interesting to integrate into this agentic BI system. Therefore, this integration of tools could need to be extended in future iterations of the DSR process. To do so, a developer must perform exactly three steps:
1. Write the new Tool's class file in metagpt/tools/, and decorate it with @register tool
61

2. Add its registered name to the tools attribute list of the BI Analytics Engineer (and also to the BI Requirements Analyst's tools attribute list if it concerns a data source system, or to the BI QA Engineer if it concerns DWH hosting)
3. Add its name to the list of available tools in the BI Solution Architect's role-level instruction prompt (so that it can be selected and assigned as the tool field on tasks in the DWH Technical Execution Plan), and if it concerns DWH hosting also to the BI QA Engineer's role-level instruction prompt (to list it as an available DWH tool)
The code content itself of each Tool class file and its inner working change from one Tool to another and basically depend on the external system or technology they connect to as well as the precise executions they perform in these systems or technologies. As this dives deeper into coding and technical implementation, which is not the main focus of this thesis, it is not developed further here.
However, it is worth noting that the mapping between one task listed in the DWH Technical Execution Plan and one call to the execute BI task() Tool dispatching function (cf. section 4.2.3.4) is not necessarily one-to-one at the level of external tool interactions. In fact, for operationally complex task types such as TRANSFORMATION tasks, executing a single task may actually require multiple sequential tool calls within the same ReAct loop (e.g. generating SQL transformation script, saving it using the Editor tool, then calling a Tool class to connect to the wanted data transformation system and to compile and execute it in that system, and finally observing the output before deciding whether to retry or continue with the next tass). Each of these steps has an observable result that informs the next reasoning step in the ReAct loop. Therefore, the Tool classes exposed to the agents in this transposed architecture are mostly written using muliple different functions rather than a single classic execute() function. By doing so, it allows the reasoning LLM behind an agent, to interact with external systems progressively across multiple ReAct loop steps. For purely mechanical operations, simple 'retry' logic can also be implemented internally within a Tool class without needing LLM involvement, then sending a structured success or fail result to the ReAct loop. This hybrid approach keeps all reasoning phases within the ReAct loop and delegates the more technical execution details to the Python layer within implemented Tool classes. This allows to stay consistent with the ReAct paradigm, while still taking into account the operational complexity of bringing automated agents in interaction with real BI tools, technologies and systems.
